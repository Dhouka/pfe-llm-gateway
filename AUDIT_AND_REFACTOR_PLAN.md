# LLM Secure Gateway — Code Audit & Refactor Plan

Prepared as a senior-engineer review of the current repository state, dated 2026-06-20.

## 1. Executive summary

The repository currently contains three overlapping implementations of the same idea (a security layer in front of Ollama) instead of one coherent system: a Spring Cloud Gateway (`gateway`, root), a Flask reimplementation of the same chat+guardrail+audit flow (`llm-flask-service`), and a third, half-integrated Flask PII microservice (`pii-service`). None of the three call each other. This is the root cause of almost everything else that feels "unfinished" — there are two parallel security policies (Java keyword lists vs. Python Presidio) that can silently drift apart, two audit logs, and no single source of truth for what "secure" means in this project.

On top of that architectural problem, the gateway itself has real gaps: the rate limiter bean exists but isn't attached to any route because `spring.cloud.gateway.routes` is empty in `application.yml`, so nothing in the README's threat model ("Redis Rate Limiting") is actually enforced end-to-end. Security policy is mostly `permitAll()`. Logging is `System.out.println` instead of a real logger. The audit trail is in-memory only and resets on every restart. Both Python services are missing dependency manifests and have a venv directory checked into git. None of this is exotic to fix, but it does mean the project is not in a state where "add a feature" is the right next move — it needs consolidation first.

The plan below is organized so you can stop after any phase and still have something more coherent than what exists today, rather than needing the full rewrite to see value.

## 2. Component-by-component assessment

**Gateway (Java/Spring, primary component).** The filter chain (`AuditFilter` → `OutputGuardrailFilter` → `PiiFilter`) is a reasonable shape for a gateway and is the right place to centralize policy. The problem is that `LlmController` doesn't trust that chain — it re-implements its own copy of the toxic/finance/injection keyword lists plus a malicious-code list the filter doesn't have, and writes its own audit entries on top of what `AuditFilter` already wrote for the same request. That means every `/llm/chat` call produces two audit rows with different shapes, and updating a guardrail phrase requires remembering to update it in two Java files. There's no actual `RouteLocator`/route definition. **Correction (confirmed during implementation):** the original claim here that Spring Cloud Gateway `GlobalFilter`s "apply globally... regardless of routes" was wrong — `GlobalFilter`s only execute for requests matched by a configured route, and since `spring.cloud.gateway.routes` was empty, `AuditFilter`/`OutputGuardrailFilter`/`PiiFilter` never ran at all, for any request. This was the single most severe bug in the repo (see issue #2, now resolved by converting all three to plain WebFlux `WebFilter`s, which register against the universal filter chain independent of routing). `LlmController` is just a normal Spring MVC/WebFlux controller calling Ollama directly via Spring AI — there are still no proxied routes, so "gateway" remains a slight misnomer for what is currently a secured REST API rather than a reverse proxy, but the security filters themselves are now actually active.

**llm-flask-service.** This duplicates the entire chat flow using Presidio instead of regex, with its own audit log, in a different language, callable on the same default Flask port (5000) as `pii-service`. It also ships test-only endpoints (`/test-toxic`, `/test-finance`, etc.) that return hardcoded canned text rather than exercising real model output — useful for demoing the dashboard, but they should be clearly marked as fixtures, not left looking like production routes.

**pii-service.** A single `/analyze` endpoint, isolated, not called by anything. Either it's the intended long-term PII engine (in which case the gateway should call it instead of regex) or it's a leftover experiment that should be deleted. Right now it's dead code from the system's point of view.

## 4. Decisions locked in

Two decisions were needed before sequencing the work, and both are now settled:

The Java gateway is the system. It becomes the only thing a client talks to. `llm-flask-service`'s chat/guardrail/audit logic is deleted; only its dashboard template survives, rebuilt to read from the gateway's `/audit` endpoint instead of its own in-memory log.

`pii-service` gets wired in rather than deleted. `PiiFilter` will call it over HTTP for ML-based detection (names, addresses, and other unstructured PII the current regexes can't catch), running alongside the existing regex checks rather than replacing them — regex stays for the structured patterns (email, card, Tunisian CIN, phone) it already handles reliably and cheaply, while Presidio handles what regex structurally can't.

This changes Phase 1 below from a deletion task into an integration task: `pii-service` needs a stable contract (request/response shape, error behavior when it's unreachable), a place in `docker-compose.yml`, and a decision on whether `PiiFilter` blocks the request if `pii-service` is down or fails open with regex-only coverage. Recommendation: fail open with a logged warning — availability of the chat feature shouldn't depend on a secondary redaction service, but that failure must be visible in logs/audit, not silent.

## 5. Phased refactor plan

**Phase 0 — Stop the bleeding (repo hygiene, ~1 day).** Remove `llm-flask-service/venv/` and all `.DS_Store` files from git, add them to `.gitignore` (currently the `.gitignore` only covers Java/IDE artifacts, nothing Python-related). Add a `requirements.txt` to both Python services pinning `flask`, `flask-cors`, `requests`, `presidio-analyzer`, `presidio-anonymizer`. Fix the port collision by giving `pii-service` its own port (e.g. 5001). None of this touches behavior, so it's safe to do first and immediately makes the repo look intentional.

**Phase 1 — Collapse the duplication.** Delete the chat/guardrail/audit logic from `llm-flask-service/app.py`, keeping only the dashboard template and a thin client that reads from the gateway's `/audit` endpoint instead of its own `audit_logs` list. Wire `pii-service` into `PiiFilter`: add a `WebClient` call to `pii-service`'s `/analyze` endpoint after the existing regex pass, merge any additional entities Presidio finds into the redacted body, give `pii-service` its own port and a `docker-compose.yml` entry, and implement fail-open behavior (log a warning, proceed with regex-only redaction) if the call errors or times out. Move the malicious-code/toxic/finance/injection keyword lists out of both `OutputGuardrailFilter` and `LlmController` into one shared `GuardrailPolicy` component (a `@Component` holding the lists plus a `classify(String)` method), and have both call it. This single change eliminates issues #1, #6, and half of #4.

**Phase 2 — Fix the audit trail.** Make `LlmController` stop writing its own `AuditEntry` and instead enrich the exchange (e.g. via a `Map` attribute on `ServerWebExchange`) with token counts/toxicity score that `AuditFilter` reads when it logs — one entry per request, one shape. Replace `CopyOnWriteArrayList` with a real persistence layer; for a banking-style compliance story, even SQLite or a Postgres table via Spring Data is a meaningful upgrade over "lost on restart." Add a correlation ID (`UUID` per request, set in `AuditFilter`, propagated via `MDC` for logging) so a single request can be traced through logs and the audit row.

**Phase 3 — Security hardening.** Replace blanket `.anyExchange().permitAll()` with an explicit allow-list (`/`, `/audit` read-only for authenticated admins only — it currently leaks every prompt's metadata to anonymous callers, which is a real problem for a project whose whole pitch is data protection). Add role/scope checks on `/llm/**` (`hasAuthority(...)`) rather than just `authenticated()`, since "logged in" and "allowed to use the LLM" aren't the same claim. Move the hardcoded Keycloak admin password and the `localhost` issuer/JWK URLs into environment variables or Spring profiles (`application-dev.yml` / `application-prod.yml`), and wire up actual Spring Cloud Gateway routes (even a single route proxying to a backend, or formalize that this is intentionally not a proxy and rename the README's claims accordingly) so the "gateway" framing is accurate and the rate limiter has something to attach to via `RequestRateLimiterGatewayFilterFactory`.

**Phase 4 — Code quality and observability.** Replace all 29 `System.out.println` calls with SLF4J (`@Slf4j` via the Lombok dependency that's already in `pom.xml` but unused) at appropriate levels (`debug` for body dumps, `warn`/`info` for blocked requests) — and make sure raw, pre-redaction bodies are never logged at a level enabled in production. Add a `@RestControllerAdvice` for consistent error JSON shape, with explicit handling for Ollama connection failures/timeouts (currently an unhandled exception will 500 with a stack trace). Add an OpenAPI dependency (`springdoc-openapi`) so `/llm/chat` and `/audit` are self-documenting.

**Phase 5 — Tests that test the real thing.** Rewrite `PiiFilterTest` and `OutputGuardrailFilterTest` to instantiate the actual `PiiFilter`/`OutputGuardrailFilter`/`GuardrailPolicy` classes (using `WebTestClient` against a minimal `WebFluxTest` context, or by extracting the regex/keyword logic into plain testable methods the filters call) instead of re-declaring the patterns inline. Add coverage for `LlmController` (mock `OllamaChatModel`, assert blocking behavior and the now-single audit entry), `RateLimiterConfig`'s security chain (test that `/llm/**` rejects unauthenticated requests and `/audit` policy matches whatever Phase 3 decides), and `AuditFilter`'s status-to-event mapping.

**Phase 6 — Packaging and docs.** Add a `Dockerfile` for the gateway so all four pieces (gateway, redis, keycloak, ollama) can run via one `docker-compose up`, instead of the gateway being the one component you run manually outside compose. Rewrite the README to describe what's actually true after the refactor — which security controls are enforced end-to-end, how to run the (now singular) chat flow, and what the audit dashboard reads from.

## 6. Suggested order of attack

Phase 0 is low-risk and worth doing regardless of which direction you go. Phase 1 is the highest-leverage change in this whole plan — it's the one that turns "three half-finished projects" into "one project," and most of the other issues either disappear or shrink once it's done. I'd recommend Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6, but Phase 3 (security) could jump ahead of Phase 2 if the audit-trail rework feels riskier than you want to take on first.

## 7. Status

Architecture decisions are locked (section 4).

Done: Phase 0 (repo hygiene), Phase 1 (collapsed duplication — shared `GuardrailPolicy`, `pii-service` wired into `PiiFilter` via `PiiServiceClient` with fail-open semantics, `llm-flask-service` stripped to dashboard + audit proxy), the critical fix from issue #2 (`GlobalFilter`→`WebFilter` conversion, see correction in section 2), Phase 2 in full (single audit entry per request via exchange attributes; `AuditLogStore` now persists each entry to a real SQLite database — `audit.db`, path configurable via `AUDIT_DB_FILE`, `PRAGMA journal_mode=WAL` — so a restart no longer loses audit history and the full history can actually be queried, e.g. `sqlite3 audit.db "select * from audit_log"`, replacing the earlier JSONL-stopgap version of this fix;
